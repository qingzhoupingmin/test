## 2. 项目初始化

### 2.1 克隆项目

```bash
git clone <仓库地址> 通用测试框架
cd 通用测试框架
```

### 2.2 目录结构

```
通用测试框架/
├── config/              ← 环境配置文件
│   ├── settings.yaml    ← 全局配置（超时、重试、日志）
│   └── env/             ← 多环境配置
│       ├── dev.yaml
│       ├── test.yaml
│       └── prod.yaml
├── data/
│   └── api/             ← 放 API 用例文件（支持 .xlsx/.csv/.json/.yaml）
├── test_cases/          ← 测试入口（不需要改）
├── hooks/               ← 自定义钩子函数
├── templates/           ← Excel 用例模板
├── utils/               ← 工具模块
├── core/                ← 框架核心引擎
├── api/                 ← HTTP 请求引擎
├── reports/             ← 报告输出目录
└── logs/                ← 日志输出目录
```

> 💡 **核心操作**：你只需在 `data/api/` 下维护用例文件（Excel/CSV/JSON/YAML 任选）+ 在 `config/` 下维护环境配置，无需写 Python 代码。

## 3. 配置说明

### 3.1 环境配置

编辑 `config/env/dev.yaml`（开发环境示例）：

```yaml
base_url: "https://dev-api.example.com"    # 被测服务地址

database:
  mysql:
    host: "${DB_HOST:localhost}"
    port: 3306
    user: "root"
    password: "${DB_PASSWORD}"              # 从环境变量注入
    database: "dev_db"
```

**多环境切换**：框架自带 `dev.yaml` / `test.yaml` / `prod.yaml` 三个环境模板，按需配置各环境的 `base_url` 和数据库地址即可。

### 3.2 全局配置

编辑 `config/settings.yaml` 调整框架行为：

```yaml
timeout:
  http_request: 30        # HTTP 请求超时秒数，可按接口响应速度调整

retry:
  max_attempts: 2         # 失败自动重试次数
  delay: 1000             # 重试间隔毫秒

logging:
  level: "INFO"           # DEBUG 可输出更多细节

hooks:
  module: "hooks.sample_hooks"   # 钩子模块路径
```

### 3.3 敏感信息处理

**绝对不要在 YAML 中写明文密码**，使用环境变量注入：

```bash
# Linux/macOS
export DB_HOST=192.168.1.100
export DB_PASSWORD=MySecret123

# Windows PowerShell
$env:DB_HOST="192.168.1.100"
$env:DB_PASSWORD="MySecret123"

# Windows CMD
set DB_HOST=192.168.1.100
set DB_PASSWORD=MySecret123
```

YAML 中使用 `${VAR:default}` 引用：
```yaml
database:
  password: "${DB_PASSWORD}"       # 必填
  host: "${DB_HOST:localhost}"     # 优先环境变量，否则用 localhost
```

## 4. 编写测试用例

> 详细模板参考 [测试用例模板.md](./测试用例模板.md)

### 4.1 API 用例快速上手

在 `data/api/` 下新建用例文件，比如 `01_用户模块.xlsx`（也可用 `.csv` / `.json` / `.yaml`）：

| case_id | case_name | method | url | body | assertions | tags |
|---|---|---|---|---|---|---|---|
| API_USER_001 | 正常登录 | POST | /api/v1/login | {"username":"admin","password":"123456"} | [{"type":"status_code","value":200},{"type":"jsonpath","key":"$.code","value":"0"}] | smoke,p0 |

**最少必填字段**：`case_id`、`case_name`、`method`、`url`、`assertions`

### 4.2 变量引用

| 写法 | 来源 | 作用域 |
|---|---|---|
| `{{token}}` | 前置用例 `extract` 提取的变量 | SESSION |
| `{{admin_username}}` | YAML 配置 `custom.admin_username` | GLOBAL |
| `${timestamp()}` | 内置函数，实时生成 | — |
| `${uuid()}` | 内置函数，实时生成 | — |

示例：在 `body` 中使用变量生成唯一测试数据：

```json
{"name": "test_${timestamp()}", "email": "${uuid()}@test.com"}
```

### 4.3 用例依赖

下游用例可通过 `depends_on` 引用上游的 `case_id`，自动获取上游提取的变量：

```
API_USER_002 (depends_on=API_USER_001)
  → 自动继承 API_USER_001 提取的 {{token}}
```

### 4.4 断言配置

断言是多条组合的 JSON 数组，常用断言类型：

```json
[
  {"type": "status_code", "value": 200},                          ← HTTP 状态码
  {"type": "jsonpath", "key": "$.code", "value": "0"},            ← JSONPath 比对
  {"type": "jsonpath", "key": "$.data.token", "not_null": true},  ← 非空校验
  {"type": "contains", "value": "success"},                       ← 响应包含文本
  {"type": "response_time", "max_ms": 3000},                      ← 响应时间
  {"type": "db", "query": "SELECT status FROM users WHERE id='{{user_id}}'", "value": "active"}  ← 数据库断言
]
```

## 5. 执行测试

### 5.1 命令行速查

```bash
# 执行全部 API 用例
pytest test_cases/test_api_executor.py --env=test --alluredir=reports/allure-results -v

# 按标签过滤（只跑冒烟测试）
pytest test_cases/test_api_executor.py --env=test -m smoke -v

# 按多个标签过滤
pytest test_cases/test_api_executor.py --env=test -m "smoke and p0" -v

# 按用例名称关键字过滤
pytest test_cases/test_api_executor.py --env=test -k "API_USER_001" -v

# 执行指定的用例文件（支持 .xlsx/.csv/.json/.yaml）
pytest test_cases/test_api_executor.py --case-file=data/api/01_用户模块.xlsx --env=test -v

# 只跑 p0 优先级的用例
pytest test_cases/test_api_executor.py --env=test -m p0 -v

# 排除慢速用例
pytest test_cases/test_api_executor.py --env=test -m "not slow" -v
```

### 5.2 并行执行

```bash
# 4 个 worker 并行（适合用例数多时加速）
pytest test_cases/ -n 4 --dist=loadscope --env=test --alluredir=reports/allure-results -v

# 自动检测 CPU 核心数
pytest test_cases/ -n auto --env=test --alluredir=reports/allure-results -v
```

### 5.3 失败重跑

```bash
# 失败用例自动重跑 2 次
pytest test_cases/ --reruns 2 --env=test --alluredir=reports/allure-results -v
```

### 5.4 调试模式

```bash
# 只执行第一个失败的用例就停止
pytest test_cases/ --env=test -x

# 显示最详细的输出
pytest test_cases/ --env=test -vv --tb=long

# 进入 pdb 调试（失败后进入交互式调试）
pytest test_cases/ --env=test --pdb
```

## 6. 查看报告

### 6.1 生成 Allure 报告

```bash
# 执行后生成 HTML 报告
allure generate reports/allure-results -o reports/allure-reports --clean

# 打开报告（自动启动本地 Web 服务）
allure open reports/allure-reports
```

### 6.2 报告内容

Allure 报告包含以下信息：

| 面板 | 内容 |
|---|---|
| **Overview** | 通过率、持续时间、严重程度分布 |
| **Suites** | 按用例文件/Sheet 分组展示用例 |
| **Behaviors** | 按功能标签（smoke/regression）分组 |
| **Graphs** | 通过率趋势图 |
| **Timeline** | 执行时间线（并行执行时有用） |
| **Environment** | 环境、分支、提交哈希等信息 |

### 6.3 单用例报告详情

点击单个用例可查看：

- ✅ 每个步骤的通过/失败状态
- ✅ 请求详情（Method / URL / Headers / Body）
- ✅ 响应详情（Status Code / Body / Response Time）
- ✅ 断言详情（期望值 vs 实际值）
- ✅ 执行日志

## 7. 高级功能

### 7.1 自定义钩子

#### 前置钩子示例：自动登录

在 `hooks/sample_hooks.py` 中定义：

```python
def login_as_admin(client, variable_manager, db=None):
    """管理员登录，获取 token 并注入到后续请求"""
    resp = client.post("/api/v1/login", json={
        "username": "admin",
        "password": "Test@123"
    })
    token = resp.json()["data"]["token"]
    variable_manager.set("token", token)
    client.session.headers["Authorization"] = f"Bearer {token}"
```

在用例的 `pre_hook` 字段填写：

```
login_as_admin
```

#### 后置钩子示例：数据清理

在 `hooks/sample_hooks.py` 中定义：

```python
def cleanup_user(client, variable_manager, db=None, result=None):
    """清理测试创建的用户"""
    user_id = variable_manager.SESSION.get("user_id")
    if user_id and db:
        db.execute(f"DELETE FROM users WHERE id = {user_id}")
```

在用例的 `post_hook` 字段填写：

```
cleanup_user
```

### 7.2 数据库断言

用例的 `assertions` 字段中直接写 SQL：

```json
[
  {"type": "status_code", "value": 200},
  {"type": "db", "query": "SELECT status FROM users WHERE id='{{user_id}}'", "value": "active"}
]
```

### 7.3 自定义函数注册

在 `core/variable_manager.py` 中注册：

```python
VariableManager.register_function("today", lambda: datetime.now().strftime("%Y-%m-%d"))
```

用例中使用：

```
{"date": "${today()}"}
```

### 7.4 鉴权方式

框架支持多种鉴权方式，通过前置钩子注入：

| 鉴权方式 | 钩子示例 |
|---|---|
| Bearer Token (JWT) | 登录后提取 token，注入 `Authorization` 头 |
| API Key | 在 header 或 query 参数中注入固定 key |
| HMAC 签名 | 按签名算法计算后注入 `X-Signature` |
| Session Cookie | 登录后的 Cookie 由 Session 自动保存 |

### 7.5 CI/CD 接入

#### Jenkins Pipeline

```groovy

    steps {
        sh '''
            pip install -r requirements.txt

```

## 8. 常见问题

### Q1: 执行报错 `unknown marker: smoke`

**原因**：pytest.ini 中未注册该 marker。

**解决**：确认 `pytest.ini` 中存在对应 marker：

```ini
markers =
    smoke: 冒烟测试
```

### Q2: 变量 `{{token}}` 未解析

**原因**：变量未被前置用例提取到变量池。

**排查**：
1. 确认上游用例的 `extract` 列正确配置（如 `token=$.data.token`）
2. 确认下游用例的 `depends_on` 指向了上游 `case_id`
3. 确认上游用例确实执行成功（提取发生在成功响应后）

### Q3: Excel 中的 JSON 解析失败

**常见错误**：
- 使用了单引号 `{'key':'value'}` → 必须改为双引号 `{"key":"value"}`
- JSON 中有换行 → 必须写成紧凑一行
- 中文引号混入 → 检查是否粘贴了 `"中文引号"`

### Q4: Allure 报告为空

**原因**：执行时未指定 `--alluredir` 参数，或目标目录权限不足。

**解决**：

```bash
# 先清理旧结果
rm -rf reports/allure-results/*

# 执行时必须加 --alluredir
pytest test_cases/ --env=test --alluredir=reports/allure-results -v

# 确认结果文件存在
ls reports/allure-results/
# 应看到 *.json 文件
```

### Q5: 并行执行时变量串了

**原因**：使用了 GLOBAL 作用域的变量，被多个线程同时读写。

**解决**：
- 需要跨用例共享的变量用 `SESSION` 作用域
- 用例内部用 `CASE` 作用域
- 使用 `${uuid()}` / `${timestamp()}` 确保数据唯一性

### Q6: 数据库断言失败

**原因**：数据库连接配置不正确或 SQL 有语法错误。

**排查**：
1. 确认 `config/env/{env}.yaml` 中的数据库配置正确
2. 确认环境变量 `DB_HOST`、`DB_PASSWORD` 已设置
3. 确认 SQL 中的变量已正确替换（使用 `{{变量名}}`）

## 9. 附录

### 9.1 CLI 参数完整列表

| 参数 | 说明 | 默认值 |
|---|---|---|
| `--env` | 环境名称（dev/test/prod） | `dev` |
| `--alluredir` | Allure 结果目录 | — |
| `--case-file` | 指定用例文件（支持多格式） | 全部 `data/api/*`（.xlsx/.csv/.json/.yaml） |
| `-m` | marker 过滤 | — |
| `-n` | 并行 worker 数 | 1（串行） |
| `--reruns` | 失败重试次数 | 0 |
| `-x` | 首个失败即停止 | — |
| `-k` | 用例名称关键字过滤 | — |
| `-v` / `-vv` | 输出详细程度 | — |

### 9.2 断言类型完整参考

| type | 参数 | 说明 | 示例 |
|---|---|---|---|
| `status_code` | `value` (int) | HTTP 状态码 | `{"type":"status_code","value":200}` |
| `jsonpath` | `key` (JSONPath), `value` 或 `not_null` | JSON 字段校验 | `{"type":"jsonpath","key":"$.code","value":"0"}` |
| `contains` | `value` (string) | 响应体包含文本 | `{"type":"contains","value":"success"}` |
| `response_time` | `max_ms` (int) | 响应时间上限 | `{"type":"response_time","max_ms":3000}` |
| `header` | `key`, `value` | 响应头校验 | `{"type":"header","key":"Content-Type","value":"application/json"}` |
| `schema` | `schema` (JSON Schema) | JSON 结构校验 | `{"type":"schema","schema":{"type":"object"}}` |
| `db` | `query` (SQL), `value` | 数据库断言 | `{"type":"db","query":"SELECT count(*) FROM users","value":1}` |
| `soft` | 同 jsonpath + `comment` | 软断言，失败不中断 | `{"type":"soft","key":"$.optional","value":"x","comment":"非关键"}` |

### 9.3 用例字段定义完整参考（适用于所有格式）

| 字段名 | 必填 | 说明 |
|---|---|---|
| `case_id` | ✅ | 唯一标识，如 `API_USER_001` |
| `case_name` | ✅ | 用例名称 |
| `module` | ❌ | 模块名（分组用） |
| `method` | ✅ | HTTP 方法：GET / POST / PUT / DELETE / PATCH |
| `url` | ✅ | 接口路径（如 `/api/v1/users`） |
| `headers` | ❌ | 请求头（JSON 格式） |
| `params` | ❌ | URL 查询参数（JSON 格式） |
| `body` | ❌ | 请求体（JSON 格式） |
| `extract` | ❌ | 变量提取规则（如 `token=$.data.token; user_id=$.data.id`） |
| `assertions` | ✅ | 断言列表（JSON Array） |
| `depends_on` | ❌ | 依赖上游 case_id（逗号分隔） |
| `pre_hook` | ❌ | 前置钩子函数名 |
| `post_hook` | ❌ | 后置钩子函数名 |
| `skip` | ❌ | Y=跳过 |
| `tags` | ❌ | 标签：smoke,regression,p0,p1,p2（逗号分隔） |
